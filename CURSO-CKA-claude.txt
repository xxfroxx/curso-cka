1. CONFIGURACION DEL ENTORNO
--------------------------------------------------

1.1 VIM
-------

El examen permite vim y nano. Vim con la configuracion
correcta te ahorra minutos evitando errores de indentacion
en YAML que son dificiles de detectar a simple vista.

Ejecutar una vez en cada nodo del laboratorio:

cat >> ~/.vimrc <<'EOF'
" --- OBLIGATORIO ---
set expandtab       " convierte Tab en espacios
                    " YAML no acepta tabs, falla silenciosamente
set tabstop=2       " un Tab equivale a 2 espacios
set shiftwidth=2    " indentacion con >> o << usa 2 espacios


" --- OPCIONAL PERO RECOMENDADO ---
set number          " muestra numeros de linea
                    " util para identificar la linea exacta
                    " cuando kubectl reporta un error de YAML


EOF

Verificacion de expandtab:
  vim /tmp/test.yaml
  Pulsa Tab en cualquier linea.
  Debe insertar 2 espacios, no el caracter ^I (tab real).

Verificacion de pastetoggle:
  Abre un fichero con vim.
  Pulsa F2 (vera "-- INSERT (paste) --" en la barra inferior).
  Pega un bloque YAML copiado.
  La indentacion debe mantenerse tal como fue copiada.
  Pulsa F2 de nuevo para volver al modo normal.

Caso real sin pastetoggle configurado:
  Copias esto desde kubernetes.io:
    spec:
      containers:
      - name: nginx
        image: nginx

  Vim lo convierte en:
    spec:
      containers:
          - name: nginx
                image: nginx

  kubectl apply falla con error de indentacion.
  Con F2 activado antes de pegar, el YAML queda intacto.

1.2 BASH: ALIASES Y VARIABLES DE ENTORNO
-----------------------------------------

Los aliases se dividen en dos grupos segun su impacto
en velocidad durante el examen.

cat >> ~/.bashrc <<'EOF'

# =============================================
# ALIASES OBLIGATORIOS - alto impacto en examen
# =============================================

alias k='kubectl'
# Ahorra ~8 caracteres por comando.
# Con 200+ comandos en 2 horas, el ahorro es significativo.

export do='--dry-run=client -o yaml'
# Genera YAML valido sin crear el recurso en el cluster.
# Base de todo trabajo imperativo. Se usa en cada pregunta.
# Uso: kubectl create deployment web --image=nginx $do

export now='--force --grace-period=0'
# Elimina pods de forma inmediata sin esperar el periodo
# de gracia (por defecto 30 segundos).
# Uso: kubectl delete pod mi-pod $now

# =============================================
# ALIASES OPCIONALES - impacto moderado
# =============================================

alias kgp='kubectl get pods'
# Util para listar pods rapidamente sin flags adicionales.
# LIMITACION: no acepta flags directamente de forma natural.
# Cuando necesites flags, usa k directamente:
#   k get pods -n kube-system      <- correcto
#   k get pods -o wide             <- correcto
#   k get pods --show-labels       <- correcto
# kgp es util solo para el caso mas simple: ver pods
# en el namespace actual sin opciones adicionales.

alias kgs='kubectl get svc'
# Mismo razonamiento que kgp pero para services.
# Para casos con flags usa k get svc directamente.

# =============================================
# AUTOCOMPLETADO - obligatorio configurar
# =============================================

source <(kubectl completion bash)
# Habilita autocompletado de kubectl para nombres de
# recursos, flags, namespaces y nombres de objetos.
# Sin esto, debes escribir cada nombre completo a mano.

complete -F __start_kubectl k
# Extiende el autocompletado al alias k.
# Sin esta linea, Tab funciona con kubectl pero no con k.

EOF

source ~/.bashrc

Verificacion de aliases:

k version --short
# debe mostrar version de cliente y servidor

k get nodes
# debe mostrar el nodo con estado Ready

kubectl create deployment demo --image=nginx $do
# debe imprimir el YAML del Deployment sin crearlo

kubectl delete pod inexistente $now 2>/dev/null; echo ok
# debe ejecutar sin esperar 30 segundos

Verificacion de autocompletado:
  Escribe "k get po" y pulsa Tab dos veces.
  Debe mostrar: pods  podtemplates  poddisruptionbudgets

NOTA SOBRE EL EXAMEN REAL:
  El autocompletado esta habilitado por defecto en el
  entorno del examen. En K3s puede requerir instalacion
  previa de bash-completion:

  apt-get install bash-completion -y
  source /usr/share/bash-completion/bash_completion
  source ~/.bashrc

2. KUBECTL IMPERATIVO - LA HERRAMIENTA CENTRAL
--------------------------------------------------

El modo imperativo de kubectl genera manifiestos YAML
validos que puedes redirigir a fichero, modificar y aplicar.
Esta tecnica es la base de velocidad en el examen.

Nunca escribas un YAML desde cero si kubectl puede
generarlo por ti.

2.1 PATRON FUNDAMENTAL
-----------------------

El patron es siempre el mismo:

  kubectl create <recurso> <nombre> <opciones> $do > fichero.yaml
  vim fichero.yaml        # editar solo si es necesario
  kubectl apply -f fichero.yaml
  kubectl get <recurso>   # verificar resultado

2.2 RECURSOS QUE APARECEN EN EL EXAMEN
----------------------------------------

Deployment:

kubectl create deployment mi-app \
  --image=nginx:1.25 \
  --replicas=3 \
  $do > deployment.yaml

cat deployment.yaml
kubectl apply -f deployment.yaml
kubectl get deployment mi-app
kubectl rollout status deployment/mi-app

Pod simple:

kubectl run mi-pod \
  --image=busybox \
  --restart=Never \
  $do > pod.yaml

kubectl apply -f pod.yaml
kubectl get pod mi-pod

Pod con comando especifico:

kubectl run debug-pod \
  --image=busybox \
  --restart=Never \
  --command \
  $do -- sh -c "sleep 3600" > pod-debug.yaml

kubectl apply -f pod-debug.yaml

Service ClusterIP (exponer un deployment existente):

kubectl expose deployment mi-app \
  --port=80 \
  --target-port=80 \
  --name=mi-app-svc \
  $do > service.yaml

kubectl apply -f service.yaml
kubectl get svc mi-app-svc

ConfigMap desde literal:

kubectl create configmap mi-config \
  --from-literal=env=produccion \
  --from-literal=debug=false \
  $do > configmap.yaml

kubectl apply -f configmap.yaml

Secret generico:

kubectl create secret generic mi-secret \
  --from-literal=password=supersecreta \
  $do > secret.yaml

kubectl apply -f secret.yaml

ServiceAccount:

kubectl create serviceaccount mi-sa $do > sa.yaml
kubectl apply -f sa.yaml

NOTA SOBRE LOS MANIFIESTOS GENERADOS:
kubectl con $do incluye campos como creationTimestamp: null
y status: {} que son inocuos. No los elimines si no es
necesario, ahorras tiempo. kubectl apply los ignora.

2.3 EXTRAER YAML DE RECURSOS EXISTENTES
-----------------------------------------

Util cuando necesitas modificar un recurso ya desplegado:

kubectl get deployment mi-app -o yaml > mi-app-backup.yaml
kubectl get pod mi-pod -o yaml
kubectl get svc mi-app-svc -o yaml

Caso real de examen: te piden cambiar la imagen de un
Deployment existente. Dos opciones:

Opcion A - edicion directa (mas rapida):
kubectl set image deployment/mi-app nginx=nginx:1.26

Opcion B - extraer, editar, aplicar (mas segura):
kubectl get deployment mi-app -o yaml > mi-app.yaml
vim mi-app.yaml   # cambiar la linea image:
kubectl apply -f mi-app.yaml

Siempre verifica despues:
kubectl rollout status deployment/mi-app
kubectl get pods -l app=mi-app

--------------------------------------------------
3. NAMESPACES Y CONTEXTOS
--------------------------------------------------

3.1 POR QUE ES CRITICO EN EL EXAMEN
--------------------------------------

El examen usa 6 clusters distintos. Cada pregunta indica
en que cluster y namespace debes trabajar. Ejecutar un
comando en el cluster o namespace equivocado es un error
que no siempre es obvio de detectar.

Error mas comun en el examen: olvidar cambiar de contexto
entre preguntas y crear recursos en el cluster anterior.

3.2 GESTION DE CONTEXTOS
--------------------------

Ver todos los contextos disponibles:

kubectl config get-contexts

La salida muestra:
  CURRENT  NAME    CLUSTER  AUTHINFO  NAMESPACE
  *        k3s     k3s      default   default

El asterisco (*) indica el contexto activo.

Cambiar de contexto:

kubectl config use-context <nombre-del-contexto>

Ver contexto actual:

kubectl config current-context

Ver configuracion completa del kubeconfig:

kubectl config view

Ver a que cluster apunta el contexto actual:

kubectl config view --minify

IMPORTANTE: En el examen, cada pregunta comienza con
el comando exacto para cambiar al contexto correcto.
Ejecutalo siempre antes de hacer cualquier otra cosa.

3.3 NAMESPACES
---------------

Crear namespace:
kubectl create namespace desarrollo

Listar namespaces:
kubectl get namespaces

Trabajar en un namespace especifico (flag -n):
kubectl get pods -n kube-system
kubectl get all -n desarrollo

Cambiar namespace por defecto en el contexto activo:

kubectl config set-context --current --namespace=desarrollo

Verificacion:
kubectl config view --minify | grep namespace

Volver al namespace default:
kubectl config set-context --current --namespace=default

TRUCO DE EXAMEN:
Si una pregunta dice "trabaja en el namespace kube-system",
ejecuta esto al inicio:

kubectl config set-context --current --namespace=kube-system

Todos los comandos siguientes no necesitan -n kube-system.
Ahorra tiempo en preguntas largas. Al terminar la pregunta,
vuelve a default para no contaminar la siguiente tarea.

--------------------------------------------------
4. INSPECCION Y DEBUGGING RAPIDO
--------------------------------------------------

Estos comandos aparecen en practicamente cada pregunta
del examen, directa o indirectamente.

4.1 DESCRIBE - LA PRIMERA HERRAMIENTA DE DIAGNOSTICO
------------------------------------------------------

kubectl describe muestra estado actual, eventos recientes
y condiciones de fallo. Es el primer comando ante cualquier
comportamiento inesperado.

kubectl describe pod <nombre>
kubectl describe node <nombre>
kubectl describe deployment <nombre>
kubectl describe svc <nombre>

Lo mas importante en la salida de describe pod:
  Seccion "Events" al final:
    muestra por que un pod falla, no arranca,
    o tiene problemas de scheduling.
  Seccion "Conditions":
    Ready, PodScheduled, ContainersReady, Initialized.
  Seccion "Containers":
    imagen actual, puertos, variables de entorno,
    volumenes montados.

4.2 LOGS - DIAGNOSTICO DE APLICACION
--------------------------------------

kubectl logs <nombre-pod>

Logs del contenedor anterior (si el pod ha reiniciado):
kubectl logs <nombre-pod> --previous

Seguir logs en tiempo real:
kubectl logs -f <nombre-pod>

Pod con multiples contenedores, especificar cual:
kubectl logs <nombre-pod> -c <nombre-contenedor>

4.3 EXEC - ENTRAR EN UN CONTENEDOR
------------------------------------

kubectl exec -it <nombre-pod> -- /bin/sh
kubectl exec -it <nombre-pod> -- /bin/bash

Ejecutar un comando sin sesion interactiva:
kubectl exec <nombre-pod> -- env
kubectl exec <nombre-pod> -- cat /etc/resolv.conf
kubectl exec <nombre-pod> -- nslookup kubernetes.default

4.4 EXTRAER CAMPOS ESPECIFICOS
--------------------------------

Tres herramientas disponibles segun el caso.
Ninguna es obligatoria memorizarla, todas son
alternativas para el mismo problema.

HERRAMIENTA 1: custom-columns
-------------------------------

Util cuando necesitas mostrar campos especificos
en formato tabla. Mas robusto que awk porque
referencia campos por nombre, no por posicion.

kubectl get pods -n kube-system \
  -o custom-columns=\
NAME:.metadata.name,\
NODE:.spec.nodeName,\
IMAGE:.spec.containers[0].image

HERRAMIENTA 2: grep/awk con --no-headers
------------------------------------------

Util para filtrar y contar resultados.

  Contar nodos Ready:
  kubectl get nodes --no-headers | grep -c Ready

  Filtrar pods que no estan Running:
  kubectl get pods --no-headers | grep -v Running

HERRAMIENTA 3: jsonpath
-------------------------

Util para extraer un valor unico y guardarlo en fichero.

  kubectl get pod mi-pod \
    -o jsonpath='{.status.podIP}' > /tmp/ip.txt

COMO DESCUBRIR LOS CAMPOS DISPONIBLES:
----------------------------------------

Sin memorizar nada. Dos comandos en terminal:

  Opcion A - ver el YAML del recurso:
  kubectl get pod <nombre> -o yaml

  La ruta en custom-columns o jsonpath es el camino
  de indentacion separado por puntos:

    spec:
      nodeName: worker-01   ->  .spec.nodeName

    spec:
      containers:
      - image: nginx        ->  .spec.containers[0].image

  Opcion B - kubectl explain para navegar la estructura:
  kubectl explain pod.spec
  kubectl explain pod.spec.containers

REGLA PARA EL EXAMEN:
----------------------

  custom-columns  -> mostrar varios campos en tabla
  grep/awk        -> filtrar y contar
  jsonpath        -> extraer un valor unico a fichero

7. LABORATORIO GUIADO - FLUJO COMPLETO
--------------------------------------------------

Este laboratorio simula el flujo de una tarea real
de examen de principio a fin.

TAREA SIMULADA:
"En el namespace 'produccion', crea un Deployment
llamado 'web-server' con imagen nginx:1.25 y 2 replicas.
Expone el Deployment con un Service ClusterIP en el
puerto 80. Guarda el numero de pods Running en /tmp/pods.txt"

Paso 1 - Cambiar al contexto y namespace correctos:

kubectl config use-context k3s
kubectl create namespace produccion
kubectl config set-context --current --namespace=produccion

Paso 2 - Crear el Deployment con modo imperativo:

kubectl create deployment web-server \
  --image=nginx:1.25 \
  --replicas=2 \
  $do > web-server.yaml

cat web-server.yaml
kubectl apply -f web-server.yaml

Paso 3 - Verificar el Deployment:

kubectl rollout status deployment/web-server
kubectl get pods -l app=web-server

Paso 4 - Crear el Service:

kubectl expose deployment web-server \
  --port=80 \
  --target-port=80 \
  $do > web-server-svc.yaml

kubectl apply -f web-server-svc.yaml
kubectl get svc web-server

Paso 5 - Guardar numero de pods Running:

kubectl get pods \
  --no-headers \
  | grep -c Running > /tmp/pods.txt

cat /tmp/pods.txt
# esperado: 2

Paso 6 - Verificacion final:

kubectl get all -n produccion

Paso 7 - Restaurar namespace default:

kubectl config set-context --current --namespace=default

8. TROUBLESHOOTING - CASOS REALES
--------------------------------------------------

CASO 1: autocompletado no funciona tras source ~/.bashrc
---------------------------------------------------------

Sintoma:
  Pulsas Tab despues de "k get po" y no ocurre nada.

Diagnostico:

  type _init_completion 2>/dev/null && echo OK || echo FALTA
  type __start_kubectl 2>/dev/null && echo OK || echo FALTA

Solucion si falta bash-completion:

  apt-get install bash-completion -y
  source /usr/share/bash-completion/bash_completion
  source ~/.bashrc

Solucion si falta kubectl completion:

  echo 'source <(kubectl completion bash)' >> ~/.bashrc
  echo 'complete -F __start_kubectl k' >> ~/.bashrc
  source ~/.bashrc

CASO 2: vim inserta tabs en lugar de espacios
----------------------------------------------

Sintoma:
  kubectl apply -f fichero.yaml devuelve:
  "error converting YAML to JSON: yaml: line X:
  found character that cannot start any token"

Diagnostico:
  cat -A fichero.yaml | head -20
  Los tabs aparecen como ^I en la salida.

Solucion inmediata dentro de vim:
  :set expandtab tabstop=2 shiftwidth=2
  :retab

  retab convierte todos los tabs existentes a espacios
  usando la configuracion de tabstop activa.

Solucion permanente:
  Verificar que ~/.vimrc tiene las lineas de la seccion 1.1.

CASO 3: YAML con indentacion duplicada al pegar
------------------------------------------------

Sintoma:
  Pegas un bloque YAML copiado desde kubernetes.io
  y la indentacion queda desfasada o duplicada.

Causa:
  El autoindent de vim suma su propia indentacion
  a la que ya trae el texto pegado.

Solucion:
  Antes de pegar, pulsa F2 para activar paste mode.
  Pega el contenido.
  Pulsa F2 de nuevo para desactivar paste mode.
  Verifica la indentacion con cat -A fichero.yaml.

CASO 4: comando ejecutado en namespace o contexto incorrecto
-------------------------------------------------------------

Sintoma:
  kubectl get pods no muestra el pod recien creado.
  O aparecen pods que no reconoces.

Diagnostico:
  kubectl config current-context
  kubectl config view --minify | grep namespace

Solucion:
  kubectl config use-context <contexto-correcto>
  kubectl config set-context --current \
    --namespace=<namespace-correcto>

Habito critico: verificar contexto y namespace activos
al inicio de cada bloque de trabajo en el examen.

--------------------------------------------------
CHECKPOINT - VERIFICACION DE COMPRENSION
--------------------------------------------------

Pregunta 1:
  Que diferencia hay entre kubectl apply y kubectl create?

Respuesta esperada:
  kubectl create falla si el recurso ya existe.
  kubectl apply crea el recurso si no existe, o actualiza
  su configuracion si ya existe. En el examen, apply es
  mas seguro como comando por defecto.

Pregunta 2:
  Como generarias el YAML de un CronJob sin crearlo
  en el cluster?

Respuesta esperada:
  kubectl create cronjob mi-job \
    --image=busybox \
    --schedule="*/5 * * * *" \
    -- echo hello \
    --dry-run=client -o yaml

Pregunta 3:
  Estas en el cluster A. La siguiente pregunta requiere
  el cluster B. Cual es el primer comando que ejecutas?

Respuesta esperada:
  kubectl config use-context <nombre-cluster-B>
  Verificar con: kubectl config current-context

Pregunta 4:
  Un pod esta en estado CrashLoopBackOff. Cuales son
  los dos primeros comandos que ejecutas?

Respuesta esperada:
  kubectl describe pod <nombre>     # ver seccion Events
  kubectl logs <nombre> --previous  # ver logs del crash

Pregunta 5:
  Como verificas que el usuario 'juan' tiene permiso
  para crear pods en el namespace 'desarrollo'?

Respuesta esperada:
  kubectl auth can-i create pods \
    --as=juan \
    -n desarrollo

LABORATORIO CRONOMETRADO
--------------------------------------------------

OBJETIVO: completar en menos de 10 minutos.
Simula el ritmo requerido en el examen real.

TAREA:

# esperado: nginx-lab  5/5  5  5

kubectl get svc nginx-lab-svc -n lab-m00
# esperado: ClusterIP  <IP>  <none>  8080/TCP

cat /tmp/imagen.txt
# esperado: nginx:latest

kubectl get pods -n lab-m00 --no-headers | grep -c Running
# esperado: 5

SOLUCION DE REFERENCIA (no mirar hasta terminar):

kubectl create namespace lab-m00

kubectl create deployment nginx-lab \
  --image=nginx:latest \
  --replicas=3 \
  -n lab-m00

kubectl expose deployment nginx-lab \
  --port=8080 \
  --target-port=80 \

1. VISION GENERAL DE LA ARQUITECTURA
--------------------------------------------------

Un cluster de Kubernetes tiene dos tipos de nodos:

CONTROL PLANE:
  El cerebro del cluster. Toma decisiones sobre
  que corre donde y mantiene el estado deseado.
  En produccion suele tener 3 o 5 nodos para alta
  disponibilidad, pero en el examen es 1 nodo.

WORKER NODES:
  Donde corren las cargas de trabajo (tus pods).
  Reciben instrucciones del control plane y las ejecutan.
  Un cluster puede tener desde 1 hasta miles de workers.

DIAGRAMA DE ARQUITECTURA:
--------------------------

  +--------------------------------------------------+
  |                  CONTROL PLANE                   |
  |                                                  |
  |  +----------+  +-----------+  +---------------+ |
  |  | kube-api |  | kube-     |  | kube-         | |
  |  | server   |  | scheduler |  | controller-   | |
  |  |          |  |           |  | manager       | |
  |  +----+-----+  +-----+-----+  +-------+-------+ |
  |       |              |                |          |
  |  +----+----+         |                |          |
  |  |  etcd   |         |                |          |
  |  | (estado)|         |                |          |
  |  +---------+         |                |          |
  |       |              |                |          |
  +-------|--------------|----------------|----------+
          |              |                |
          | API calls    | watch/update   | watch/update
          |              |                |
  +-------|--------------|----------------|----------+
  |       |         WORKER NODE           |          |
  |  +----+-----+                         |          |
  |  |  kubelet |<------------------------+          |
  |  |          |                                    |
  |  +----+-----+                                    |
  |       |                                          |
  |  +----+----------+  +--------------------+       |
  |  | container     |  | kube-proxy         |       |
  |  | runtime       |  | (reglas de red)    |       |
  |  | (containerd)  |  +--------------------+       |
  |  +---------------+                               |
  |                                                  |
  |  +-------+  +-------+  +-------+                |
  |  | Pod A |  | Pod B |  | Pod C |                |
  |  +-------+  +-------+  +-------+                |
  +--------------------------------------------------+

--------------------------------------------------
2. COMPONENTES DEL CONTROL PLANE
--------------------------------------------------

2.1 kube-apiserver
-------------------

Es el unico punto de entrada al cluster. Todo pasa
por aqui: kubectl, otros componentes, y herramientas
externas. Nadie escribe en etcd directamente, solo
el apiserver.

Que hace exactamente:
  - Recibe peticiones REST (kubectl apply, get, delete)
  - Autentica y autoriza cada peticion (RBAC)
  - Valida que el objeto YAML tenga la estructura correcta
  - Persiste el estado en etcd
  - Notifica a los componentes que observan cambios

Donde vive en kubeadm:
  Static pod en /etc/kubernetes/manifests/kube-apiserver.yaml
  Logs: kubectl logs -n kube-system kube-apiserver-<nodo>

Donde vive en K3s:
  Proceso integrado dentro del binario k3s server
  Logs: journalctl -u k3s

Puerto por defecto: 6443 (HTTPS)

Verificacion:
  curl -k https://localhost:6443/healthz
  # esperado: ok

2.2 etcd
---------

Base de datos clave-valor distribuida donde Kubernetes
guarda TODO el estado del cluster: pods, services,
configmaps, secrets, roles, bindings, etc.

Es el componente mas critico del cluster. Si etcd
muere o se corrompe, el cluster pierde su estado.
Por eso el examen CKA incluye backup y restore de etcd.

Que hace exactamente:
  - Almacena el estado deseado y actual de todos los objetos
  - Usa el protocolo Raft para consenso distribuido
    (relevante solo en clusters de alta disponibilidad)
  - Notifica al apiserver cuando un valor cambia (watch)

Donde vive en kubeadm:
  Static pod en /etc/kubernetes/manifests/etcd.yaml
  Datos en /var/lib/etcd/
  Logs: kubectl logs -n kube-system etcd-<nodo>

Donde vive en K3s:
  Reemplazado por SQLite por defecto.
  Datos en /var/lib/rancher/k3s/server/db/
  Esto significa que en K3s NO puedes usar etcdctl.
  Para el examen real usaras kubeadm donde etcd es real.

Verificacion en kubeadm:
  ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key \
    endpoint health

2.3 kube-scheduler
-------------------

Observa pods que no tienen nodo asignado y decide
en que nodo deben correr. No ejecuta el pod, solo
escribe el nombre del nodo en el campo spec.nodeName
del pod. El kubelet del nodo destino hace el resto.

Que hace exactamente:
  - Filtra nodos que no cumplen los requisitos del pod
    (recursos, taints, affinity, etc.)
  - Puntua los nodos que si cumplen los requisitos
  - Asigna el pod al nodo con mayor puntuacion

Donde vive en kubeadm:
  Static pod en /etc/kubernetes/manifests/kube-scheduler.yaml
  Logs: kubectl logs -n kube-system kube-scheduler-<nodo>

Sintoma de fallo:
  Pods se quedan en estado Pending indefinidamente.
  kubectl describe pod muestra Events vacios o
  "no nodes available to schedule pods".

2.4 kube-controller-manager
-----------------------------

Ejecuta los controladores que mantienen el estado
deseado del cluster. Un controlador es un bucle
que observa el estado actual, lo compara con el
estado deseado y toma acciones para reconciliarlos.

Controladores principales que incluye:
  - Deployment controller: mantiene el numero de replicas
  - Node controller: detecta nodos caidos
  - Job controller: gestiona la ejecucion de Jobs
  - ServiceAccount controller: crea SAs por defecto
  - Endpoints controller: actualiza endpoints de services

Donde vive en kubeadm:
  Static pod en /etc/kubernetes/manifests/kube-controller-manager.yaml
  Logs: kubectl logs -n kube-system kube-controller-manager-<nodo>

Sintoma de fallo:
  Deployments no crean pods aunque el scheduler funcione.
  Nodos caidos no se marcan como NotReady.

--------------------------------------------------
3. COMPONENTES DEL WORKER NODE
--------------------------------------------------

3.1 kubelet
------------

El agente principal en cada nodo. Es el unico
componente que NO corre como pod, corre directamente
como servicio del sistema operativo.

Que hace exactamente:
  - Se registra en el apiserver como nodo disponible
  - Observa los pods asignados a su nodo via apiserver
  - Instruye al container runtime para arrancar/parar
    contenedores
  - Reporta el estado de los pods y del nodo al apiserver
  - Ejecuta liveness y readiness probes
  - Gestiona los static pods (ver seccion 5)

Donde vive:
  Servicio systemd en todos los nodos (kubeadm y K3s)
  Configuracion: /var/lib/kubelet/config.yaml
  Logs: journalctl -u kubelet -f

Sintoma de fallo:
  Nodo aparece como NotReady en kubectl get nodes.
  Los pods del nodo dejan de reportar estado.

Verificacion:
  systemctl status kubelet
  journalctl -u kubelet --no-pager | tail -20

3.2 kube-proxy
---------------

Corre en cada nodo y mantiene las reglas de red
que permiten que los Services funcionen.

Que hace exactamente:
  - Observa Services y Endpoints en el apiserver
  - Traduce la IP virtual del Service (ClusterIP) en
    reglas de iptables o ipvs que redirigen trafico
    a los pods correspondientes
  - No maneja trafico directamente, solo configura
    las reglas del kernel

Donde vive:
  DaemonSet en el namespace kube-system
  kubectl get pods -n kube-system | grep kube-proxy

Sintoma de fallo:
  Los Services no funcionan aunque los pods esten Running.
  Conexiones a ClusterIP no llegan a los pods.

3.3 container runtime
----------------------

El software que realmente ejecuta los contenedores.
Kubernetes habla con el runtime via CRI
(Container Runtime Interface).

Runtime mas comun: containerd
Otros: CRI-O, Docker (deprecado en versiones recientes)

Verificacion:
  crictl ps          # listar contenedores corriendo
  crictl images      # listar imagenes en el nodo

Donde vive:
  Servicio systemd: systemctl status containerd

--------------------------------------------------
4. FLUJO COMPLETO: DEL KUBECTL AL CONTENEDOR
--------------------------------------------------

Este es el flujo que ocurre cuando ejecutas:
kubectl apply -f deployment.yaml

DIAGRAMA DEL FLUJO:
--------------------

  kubectl apply
       |
       | 1. HTTP POST /apis/apps/v1/namespaces/default/deployments
       v
  kube-apiserver
       |
       | 2. Autentica, autoriza, valida el objeto
       | 3. Persiste el Deployment en etcd
       v
  etcd
       |
       | 4. Notifica cambio al apiserver (watch)
       v
  kube-controller-manager (Deployment controller)
       |
       | 5. Detecta que el Deployment necesita 3 pods
       | 6. Crea 3 objetos Pod en etcd via apiserver
       v
  kube-scheduler
       |
       | 7. Detecta 3 pods sin nodo asignado
       | 8. Evalua nodos disponibles
       | 9. Asigna spec.nodeName a cada pod
       v
  kubelet (en el nodo asignado)
       |
       | 10. Detecta pod asignado a su nodo
       | 11. Instruye a containerd para descargar imagen
       | 12. Arranca el contenedor
       | 13. Reporta estado Running al apiserver
       v
  Pod en estado Running

PASO A PASO DETALLADO:
-----------------------

Paso 1 - kubectl envia la peticion:
  kubectl serializa el YAML a JSON y hace un HTTP POST
  al apiserver en el puerto 6443.

Paso 2 - apiserver autentica la peticion:
  Verifica el certificado o token del cliente.
  Comprueba en RBAC si el usuario tiene permiso
  para crear Deployments en ese namespace.

Paso 3 - apiserver valida el objeto:
  Comprueba que el YAML tiene todos los campos
  obligatorios y que los valores son correctos.

Paso 4 - apiserver persiste en etcd:
  Guarda el objeto Deployment con estado "creado".

Paso 5 - Deployment controller actua:
  El controller-manager tiene un watch sobre Deployments.
  Detecta el nuevo Deployment y calcula cuantos pods
  necesita crear para cumplir spec.replicas.

Paso 6 - Se crean los objetos Pod:
  El controller crea los pods via apiserver.
  En este momento los pods existen en etcd pero
  no tienen nodo asignado (spec.nodeName vacio).

Paso 7 - Scheduler actua:
  El scheduler tiene un watch sobre pods sin nodeName.
  Detecta los nuevos pods y comienza el proceso
  de seleccion de nodo.

Paso 8 - Scheduler filtra y puntua nodos:
  Filtra nodos que no tienen recursos suficientes,
  que tienen taints que el pod no tolera, etc.
  Puntua los nodos restantes y elige el mejor.

Paso 9 - Scheduler asigna el nodo:
  Escribe spec.nodeName en el objeto Pod via apiserver.
  El scheduler no hace nada mas, su trabajo termina aqui.

Paso 10 - kubelet detecta el pod:
  El kubelet de cada nodo tiene un watch sobre pods
  donde spec.nodeName coincide con su nombre de nodo.
  Detecta el nuevo pod asignado.

Paso 11 - kubelet instruye al runtime:
  Llama a containerd via CRI para descargar la imagen
  si no esta en cache local.

Paso 12 - containerd arranca el contenedor:
  Descarga la imagen del registry.
  Crea el contenedor y lo arranca.

Paso 13 - kubelet reporta el estado:
  Actualiza el estado del pod a Running en el apiserver.
  A partir de aqui ejecuta liveness y readiness probes
  periodicamente.

--------------------------------------------------
5. STATIC PODS
--------------------------------------------------

Los componentes del control plane en kubeadm corren
como static pods. Este es un concepto critico para
el examen porque aparece en troubleshooting del
control plane.

QUE ES UN STATIC POD:
----------------------

Un pod que el kubelet arranca directamente leyendo
un fichero YAML de un directorio local, sin
intervension del apiserver ni del scheduler.

El kubelet monitoriza ese directorio y:
  - Si aparece un fichero YAML, arranca el pod
  - Si se modifica el fichero, reinicia el pod
  - Si se elimina el fichero, detiene el pod

DIRECTORIO DE STATIC PODS EN KUBEADM:
---------------------------------------

  /etc/kubernetes/manifests/

  Contenido tipico:
  etcd.yaml
  kube-apiserver.yaml
  kube-controller-manager.yaml
  kube-scheduler.yaml

COMO MODIFICAR UN COMPONENTE DEL CONTROL PLANE:
-------------------------------------------------

Para cambiar la configuracion del apiserver, por ejemplo:

  vim /etc/kubernetes/manifests/kube-apiserver.yaml

  El kubelet detecta el cambio y reinicia el pod
  automaticamente en segundos.

  Verificacion:
  kubectl get pods -n kube-system | grep apiserver
  # el pod debe reiniciarse y volver a Running

POR QUE ESTO ES CRITICO EN EL EXAMEN:
---------------------------------------

El examen puede pedirte:
  - Modificar un flag del apiserver
  - Diagnosticar por que un componente del control plane
    no arranca (fichero YAML corrupto en manifests/)
  - Restaurar un componente eliminado accidentalmente

NOTA SOBRE K3s:
----------------

K3s NO usa static pods para sus componentes internos.
Todo corre dentro del proceso k3s server como goroutines.
No hay directorio /etc/kubernetes/manifests/ en K3s.
Para practicar static pods necesitas el entorno kubeadm.

--------------------------------------------------
6. COMUNICACION Y CERTIFICADOS
--------------------------------------------------

Todos los componentes de Kubernetes se comunican
por HTTPS con certificados TLS mutuos (mTLS).
Esto significa que tanto el cliente como el servidor
presentan certificados para autenticarse.

MAPA DE CERTIFICADOS EN KUBEADM:
----------------------------------

  /etc/kubernetes/pki/

  Ficheros principales:
  ca.crt / ca.key          -> CA raiz del cluster
  apiserver.crt/.key       -> certificado del apiserver
  apiserver-kubelet-client.crt/.key
                           -> apiserver habla con kubelet
  etcd/ca.crt              -> CA especifica de etcd
  etcd/server.crt/.key     -> certificado del servidor etcd

KUBECONFIG - COMO KUBECTL SE AUTENTICA:
-----------------------------------------

El fichero ~/.kube/config contiene:
  - La URL del apiserver
  - El certificado de cliente del usuario
  - El certificado de la CA para verificar el servidor

Ver la configuracion activa:
  kubectl config view

Ver el certificado del usuario actual:
  kubectl config view --raw \
    | grep client-certificate-data \
    | awk '{print $2}' \
    | base64 -d \
    | openssl x509 -text -noout \
    | grep -A2 Subject

8. LABORATORIO GUIADO
--------------------------------------------------

Objetivo: mapear la arquitectura real de tu K3s
y entender donde vive cada componente.

Paso 1 - Ver los componentes del control plane:

kubectl get pods -n kube-system

Identifica en la salida: coredns, traefik (ingress),
local-path-provisioner. En K3s estos reemplazan o
complementan los componentes estandar.

Paso 2 - Ver todos los nodos y su rol:

kubectl get nodes -o wide

La columna ROLES muestra: control-plane, worker o
ambos (en K3s single-node es control-plane,master).

Paso 3 - Inspeccionar el nodo en detalle:

kubectl describe node <nombre-del-nodo>

Busca estas secciones:
  - Conditions: Ready, MemoryPressure, DiskPressure
  - Capacity vs Allocatable: recursos totales vs disponibles
  - System Info: version de kubelet, container runtime
  - Allocated resources: cuanto esta en uso

Paso 4 - Ver el estado del kubelet:

systemctl status k3s
# En K3s el kubelet corre dentro del proceso k3s

journalctl -u k3s --no-pager | tail -30

Paso 5 - Seguir el flujo completo del apartado 4:

kubectl apply -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: flujo-test
  namespace: default
spec:
  replicas: 2
  selector:
    matchLabels:
      app: flujo-test
  template:
    metadata:
      labels:
        app: flujo-test
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        resources:
          requests:
            memory: "64Mi"
            cpu: "50m"
          limits:
            memory: "128Mi"
            cpu: "100m"
EOF

Observar el scheduler asignando nodos:
kubectl get pods -l app=flujo-test -o wide -w

La flag -w (watch) muestra cambios en tiempo real.
Veras la progresion: Pending -> ContainerCreating -> Running

Paso 6 - Ver en que nodo corren y verificar asignacion:

kubectl get pods -l app=flujo-test \
  -o custom-columns=\
NAME:.metadata.name,\
NODE:.spec.nodeName,\
STATUS:.status.phase,\
IP:.status.podIP

Paso 7 - Ver los eventos del scheduler:

kubectl get events --sort-by='.lastTimestamp' | tail -10

Busca eventos de tipo "Scheduled" que muestran
la decision del scheduler.

Paso 8 - Limpiar:

kubectl delete deployment flujo-test

--------------------------------------------------
9. TROUBLESHOOTING - CASOS REALES
--------------------------------------------------

CASO 1: nodo en estado NotReady
---------------------------------

Sintoma:
  kubectl get nodes muestra un nodo como NotReady.

Diagnostico paso a paso:

  Paso 1 - Confirmar el estado y cuanto tiempo lleva:
  kubectl get nodes
  kubectl describe node <nombre-nodo>
  # buscar seccion Conditions, campo Ready

  Paso 2 - Ver el motivo en los eventos del nodo:
  kubectl describe node <nombre-nodo> | grep -A5 Conditions

  Paso 3 - Verificar el kubelet en el nodo afectado:
  # conectarse al nodo via SSH
  ssh usuario@<ip-nodo>
  systemctl status kubelet
  journalctl -u kubelet --no-pager | tail -50

  Causas mas comunes:
  - kubelet detenido: systemctl start kubelet
  - kubelet no puede contactar apiserver: problema de red
    o certificados expirados
  - Disco lleno: df -h, limpiar imagenes con crictl rmi
  - Memoria agotada: el OOM killer puede matar el kubelet

  Paso 4 - Verificar conectividad al apiserver:
  curl -k https://<ip-control-plane>:6443/healthz

CASO 2: pod en Pending sin razon aparente
------------------------------------------

Sintoma:
  Un pod lleva minutos en Pending.
  kubectl get pods muestra Pending pero no da detalles.

Diagnostico paso a paso:

  Paso 1 - Ver los eventos del pod:
  kubectl describe pod <nombre>
  # seccion Events al final, es la mas importante

  Paso 2 - Interpretar los mensajes de Events:

  "0/1 nodes are available: 1 Insufficient memory"
  -> el pod pide mas memoria de la disponible
  -> solucion: reducir resources.requests.memory

  "0/1 nodes are available: 1 node(s) had taint"
  -> el nodo tiene un taint que el pod no tolera
  -> solucion: agregar toleracion al pod o
     eliminar el taint del nodo

  "0/1 nodes are available: 1 node(s) didn't match
   node affinity/selector"
  -> el nodeSelector o affinity no coincide con
     ningun nodo disponible
  -> solucion: corregir los labels del nodo o
     el nodeSelector del pod

  Paso 3 - Ver recursos disponibles en los nodos:
  kubectl describe node <nombre> | grep -A5 "Allocated resources"

CASO 3: componente del control plane no arranca
------------------------------------------------

Sintoma (en kubeadm):
  kubectl get pods -n kube-system muestra el
  kube-scheduler en CrashLoopBackOff o no aparece.

Diagnostico paso a paso:

  Paso 1 - Ver el estado del pod:
  kubectl get pods -n kube-system
  kubectl describe pod kube-scheduler-<nodo> -n kube-system

  Paso 2 - Ver los logs del componente:
  kubectl logs kube-scheduler-<nodo> -n kube-system

  Si el pod no existe o no responde, ir directamente
  a los logs del kubelet que gestiona el static pod:
  journalctl -u kubelet | grep scheduler | tail -20

  Paso 3 - Verificar el fichero de static pod:
  cat /etc/kubernetes/manifests/kube-scheduler.yaml

  Causas mas comunes:
  - YAML corrupto (error de indentacion en el manifiesto)
  - Flag desconocido en el comando de arranque
  - Certificado expirado o ruta incorrecta
  - Puerto en uso por otro proceso

  Paso 4 - Reparar el manifiesto:
  vim /etc/kubernetes/manifests/kube-scheduler.yaml
  # corregir el error
  # el kubelet detecta el cambio y reinicia el pod
  # en 10-20 segundos

  Verificacion:
  kubectl get pods -n kube-system -w
  # esperar a que el pod vuelva a Running

--------------------------------------------------
CHECKPOINT - VERIFICACION DE COMPRENSION
--------------------------------------------------

Pregunta 1:
  Un pod lleva 5 minutos en Pending. Cual es el
  primer componente que debes investigar y por que?

Respuesta esperada:
  El kube-scheduler. El estado Pending significa que
  el pod existe en etcd pero no tiene nodo asignado.
  El scheduler es el responsable de esa asignacion.
  Si el scheduler falla, todos los pods nuevos
  quedan en Pending indefinidamente.

Pregunta 2:
  Que diferencia hay entre un static pod y un pod
  normal? Quien es responsable de cada uno?

Respuesta esperada:
  Un pod normal es gestionado por el apiserver y
  el scheduler. Un static pod es gestionado
  directamente por el kubelet, que lo arranca
  leyendo un fichero YAML de un directorio local
  (/etc/kubernetes/manifests/ en kubeadm).
  Los componentes del control plane son static pods.

Pregunta 3:
  Si el apiserver deja de funcionar, que componentes
  siguen funcionando y cuales se detienen?

Respuesta esperada:
  Siguen funcionando:
  - Los pods que ya estan corriendo (el kubelet
    los mantiene aunque no pueda contactar al apiserver)
  - El container runtime (containerd)

  Se detienen o degradan:
  - kubectl (no puede conectar al apiserver)
  - El scheduler (no puede leer ni escribir pods)
  - El controller-manager (no puede actuar)
  - No se pueden crear, modificar ni eliminar recursos

Pregunta 4:
  Donde estan los logs del kubelet y como los ves
  en tiempo real?

Respuesta esperada:
  El kubelet corre como servicio systemd.
  Sus logs estan en journald.
  Para verlos en tiempo real:
  journalctl -u kubelet -f

Pregunta 5:
  En K3s, como diagnosticas un problema del
  control plane si no hay static pods?

Respuesta esperada:
  En K3s todos los componentes del control plane
  corren dentro del proceso k3s server.
  Los logs de todos estan en:
  journalctl -u k3s -f
  No hay pods individuales por componente que inspeccionar.

PRERREQUISITOS DEL SISTEMA OPERATIVO
--------------------------------------------------

Antes de instalar cualquier componente de Kubernetes,
el sistema operativo debe estar correctamente
configurado. Estos pasos aplican a los 3 nodos.

Conexion de M01: en arquitectura vimos que kubelet
y kube-proxy necesitan que el kernel reenvie trafico
entre interfaces de red. Estos parametros lo habilitan.

PASO 1 - MODULOS DE KERNEL [CP+W]
-----------------------------------

Los modulos overlay y br_netfilter son requeridos
por containerd y por el networking de Kubernetes.

  overlay:      permite a containerd usar overlayfs
                como storage driver para las capas
                de las imagenes de contenedor
  br_netfilter: permite que iptables inspeccione
                trafico que pasa por bridges de red
                necesario para NetworkPolicy y kube-proxy

cat <<EOF | sudo tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF

sudo modprobe overlay
sudo modprobe br_netfilter

Verificacion:
lsmod | grep overlay
lsmod | grep br_netfilter
# ambos deben aparecer en la salida

PASO 2 - PARAMETROS SYSCTL [CP+W]
-----------------------------------

Estos parametros permiten que iptables procese
trafico de red de los pods correctamente:

  bridge-nf-call-iptables:  trafico IPv4 en bridges
                             pasa por iptables
  bridge-nf-call-ip6tables: mismo para IPv6
  ip_forward:               permite al nodo reenviar
                             paquetes entre interfaces
                             necesario para pod-to-pod

cat <<EOF | sudo tee /etc/sysctl.d/k8s.conf
net.bridge.bridge-nf-call-iptables = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward = 1
EOF

sudo sysctl --system

Validacion:
sysctl net.bridge.bridge-nf-call-iptables
sysctl net.bridge.bridge-nf-call-ip6tables
sysctl net.ipv4.ip_forward
# resultado esperado en los 3:
# net.bridge.bridge-nf-call-iptables = 1
# net.bridge.bridge-nf-call-ip6tables = 1
# net.ipv4.ip_forward = 1

INSTALACION DE CONTAINERD Y KUBETOOLS
--------------------------------------------------

Para replicar el entorno del examen CKA se usan
dos scripts que automatizan la instalacion:

setup-container.sh:
  - Instala containerd desde GitHub releases
  - Configura SystemdCgroup = true en containerd
    (obligatorio para que kubelet funcione)
  - Instala runc desde opencontainers
  - Aplica fix de apparmor especifico de Ubuntu 24.04

setup-kubetools-previousversion.sh:

INICIALIZACION DEL CLUSTER CON KUBEADM
--------------------------------------------------

COMO FUNCIONA KUBEADM INIT INTERNAMENTE:
-----------------------------------------

Conexion con M01: en arquitectura vimos los
componentes del control plane. kubeadm init
los despliega en este orden exacto:

  1. Preflight checks
     Verifica que el sistema cumple los requisitos:
     swap desactivado, puertos libres, containerd
     corriendo, permisos de root, etc.

  2. Genera la CA (Certificate Authority)
     Crea los certificados TLS en /etc/kubernetes/pki/
     Cada componente tiene su propio certificado
     firmado por esta CA.

  3. Genera kubeconfig files
     /etc/kubernetes/admin.conf     (kubectl admin)
     /etc/kubernetes/kubelet.conf   (kubelet del CP)
     /etc/kubernetes/scheduler.conf (kube-scheduler)
     /etc/kubernetes/controller-manager.conf

  4. Despliega los static pods del control plane
     /etc/kubernetes/manifests/etcd.yaml
     /etc/kubernetes/manifests/kube-apiserver.yaml
     /etc/kubernetes/manifests/kube-controller-manager.yaml
     /etc/kubernetes/manifests/kube-scheduler.yaml

     Estos ficheros son leidos directamente por kubelet.
     No necesitan kubectl ni la API para arrancar.
     Por eso el control plane puede inicializarse
     sin un cluster previo.

  5. Espera a que la API este disponible
     Hace polling hasta que kube-apiserver responde.

  6. Configura RBAC inicial
     Crea los ClusterRoles y ClusterRoleBindings
     necesarios para el funcionamiento del cluster.

  7. Genera el token de bootstrap
     Para que los workers puedan unirse de forma segura.

  8. Instala kube-proxy y CoreDNS como addons
     Como pods normales via la API ya operativa.

EJECUTAR EN cka-cp1 UNICAMENTE:
---------------------------------

  sudo kubeadm init

GUARDAR TODA LA SALIDA. Incluye el kubeadm join
que necesitaras para los workers.

Configurar kubectl tras el init:

  mkdir -p $HOME/.kube
  sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
  sudo chown $(id -u):$(id -g) $HOME/.kube/config

Verificacion:

  kubectl get nodes
  # esperado: cka-cp1   NotReady   control-plane
  # NotReady es correcto, aun no hay CNI instalado

  kubectl get pods -n kube-system
  # etcd, apiserver, controller-manager, scheduler
  # deben estar Running
  # coredns estara Pending hasta instalar el CNI

INSTALACION DE CALICO (CNI)

POR QUE SE INSTALA DESPUES DE KUBEADM INIT:

COMO FUNCIONA CALICO INTERNAMENTE:
------------------------------------

Conexion con M01: en arquitectura vimos que los
pods necesitan una red para comunicarse entre nodos.
El CNI es el componente que la provee.

Calico despliega en cada nodo:
  calico-node (DaemonSet):

    - Programa las reglas de iptables para pods
    - Implementa NetworkPolicy

  calico-kube-controllers (Deployment):
    - Sincroniza el estado de Calico con la API
    - Gestiona el ciclo de vida de los recursos

  # calico-kube-controllers esten Running

--------------------------------------------------
UNION DE WORKERS AL CLUSTER
--------------------------------------------------

COMO FUNCIONA KUBEADM JOIN INTERNAMENTE:
------------------------------------------

  1. El worker contacta al API server del CP
     usando el token de bootstrap generado en el init

  2. El API server valida el token y devuelve
     la informacion del cluster (CA, endpoint)

  3. El worker genera su propio certificado kubelet
     firmado por la CA del cluster

  4. kubelet en el worker se registra en la API
     como un nuevo Node object

  5. El scheduler puede ahora asignar pods al worker

EJECUTAR EN cka-worker1 Y cka-worker2:

  El comando exacto lo obtienes de la salida de
  kubeadm init. Tiene esta forma:

  sudo kubeadm join 172.16.46.130:6443 \
    --token <token> \
    --discovery-token-ca-cert-hash sha256:<hash>

  Si perdiste la salida de kubeadm init:
  kubeadm token create --print-join-command

Verificacion desde cka-cp1 tras unir ambos workers:

  kubectl get nodes -o wide
  # esperado:
  # NAME          STATUS   ROLES           VERSION
  # cka-cp1       Ready    control-plane   v1.35.5
  # cka-worker1   Ready    <none>          v1.35.5
  # cka-worker2   Ready    <none>          v1.35.5

  kubectl get pods -A
  # todos los pods deben estar Running

--------------------------------------------------
ETCD - BACKUP Y RESTORE
--------------------------------------------------

IMPORTANCIA EN EL EXAMEN:
---------------------------

El backup y restore de etcd es una de las tareas
mas frecuentes y con mayor peso en el examen CKA.
Aparece en practicamente todos los examenes.

Debes poder ejecutarlo en menos de 5 minutos.

QUE ES ETCD Y POR QUE SE HACE BACKUP:
----------------------------------------

Conexion con M01: etcd es la base de datos del
cluster. Todo el estado de Kubernetes vive ahi:
pods, deployments, services, secrets, configmaps,
RBAC, etc.

Si etcd se corrompe o pierde datos, el cluster
pierde todo su estado. No hay pods, no hay nada.
El backup permite restaurar ese estado.


PASOS DEL CURSO:
======================
sudo apt install etcd-client
sudo etcdctl --help; sudo ETCDCTL_API=3 etcdctl --help
ps aux | grep etcd

Test API access:
sudo ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  get / --prefix --keys-only

Make the backup: sudo ETCDCTL_API=3 etcdctl --endpoints=localhost:2379 --cacert /etc/kubernetes/pki/etcd/ca.crt --cert
/etc/kubernetes/pki/etcd/server.crt --key
/etc/kubernetes/pki/etcd/server.key snapshot save /tmp/etcdbackup.dbDemo: 

Verifying the Etcd Backup

sudo ETCDCTL_API=3 etcdctl --write-out=table snapshot status
/tmp/etcdbackup.db
Just to be sure: cp /tmp/etcdbackup.db /tmp/etcdbackup.db.2
======================

INSTALACION DE ETCDCTL:
-------------------------

etcdctl es el cliente de linea de comandos para
interactuar con etcd directamente.

  sudo apt install etcd-client

  etcdctl version
  # etcdctl version: 3.4.30
  # API version: 3.4

COMO LOCALIZAR LOS CERTIFICADOS DE ETCD:
------------------------------------------

etcdctl necesita autenticarse con etcd usando TLS.
Los certificados estan en /etc/kubernetes/pki/etcd/

  ls /etc/kubernetes/pki/etcd/
  # ca.crt        CA de etcd
  # server.crt    certificado del servidor etcd
  # server.key    clave privada del servidor etcd

Tambien puedes verlos en el manifiesto de etcd:
  cat /etc/kubernetes/manifests/etcd.yaml | grep file

BACKUP DE ETCD:
----------------

  sudo ETCDCTL_API=3 etcdctl snapshot save \
    /opt/etcd-backup.db \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key

Verificar que el backup se creo correctamente:

  sudo ETCDCTL_API=3 etcdctl snapshot status \
    /opt/etcd-backup.db \
    --write-out=table

  Salida esperada:
  +----------+----------+------------+------------+
  |   HASH   | REVISION | TOTAL KEYS | TOTAL SIZE |
  +----------+----------+------------+------------+
  | xxxxxxxx |    XXXX  |    XXX     |   X.X MB   |
  +----------+----------+------------+------------+

  ls -lh /opt/etcd-backup.db
  # verificar que el fichero existe y tiene tamano > 0

RESTORE DE ETCD:
-----------------

El restore es el proceso mas delicado y el que
mas errores causa en el examen. Sigue este orden
exacto sin saltarte pasos.

POR QUE EL ORDEN IMPORTA:
  etcd no puede restaurarse mientras esta corriendo.
  Como corre como static pod, debes moverlo
  temporalmente fuera de /etc/kubernetes/manifests/
  para que kubelet lo detenga.

TIP antes del restore, sacar los valores en uso así:

sudo grep -E -- '--name=|--initial-cluster=|--initial-advertise-peer-urls=|--listen-client-urls=|--advertise-client-urls=|--listen-peer-urls=' /etc/kubernetes/manifests/etcd.yaml

Paso 1 - Crear los datos restaurados en una nueva ruta:

  sudo ETCDCTL_API=3 etcdctl snapshot restore \
    /opt/etcd-backup.db \
    --data-dir=/var/lib/etcd-restored

  Esto crea un nuevo directorio con los datos
  restaurados. No sobreescribe etcd en caliente.

Paso 2 - Detener etcd moviendo el manifiesto:

  sudo mv /etc/kubernetes/manifests/etcd.yaml \
    /tmp/etcd.yaml

  kubelet detecta el cambio y detiene el pod de etcd
  en unos segundos. Verificar:

  watch crictl ps | grep etcd
  # esperar hasta que etcd desaparezca de la lista
  # Ctrl+C

Paso 3 - Reemplazar el directorio de datos de etcd:

  sudo mv /var/lib/etcd /var/lib/etcd-old
  sudo mv /var/lib/etcd-restored /var/lib/etcd

Paso 4 - Restaurar el manifiesto de etcd:

  sudo mv /tmp/etcd.yaml \
    /etc/kubernetes/manifests/etcd.yaml

  kubelet detecta el manifiesto y arranca etcd
  con los datos restaurados. Puede tardar 1-2 minutos.

Paso 5 - Verificar que etcd y el cluster se recuperaron:

  watch crictl ps | grep etcd
  # esperar hasta que etcd aparezca como Running
  # Ctrl+C

  kubectl get nodes
  # el cluster debe responder correctamente

  kubectl get pods -A
  # todos los pods del sistema deben estar Running

TRUCO DE EXAMEN:
  Si kubectl no responde inmediatamente tras el restore,
  espera 60 segundos. etcd necesita tiempo para
  inicializarse completamente antes de que el
  apiserver pueda conectarse.

--------------------------------------------------
UPGRADE DEL CLUSTER
--------------------------------------------------

IMPORTANCIA EN EL EXAMEN:
---------------------------

El upgrade es otra tarea frecuente en el examen.
El proceso tiene un orden estricto que no puedes
alterar:

  1. Siempre upgrade del control plane primero
  2. Luego upgrade de los workers uno a uno
  3. Nunca saltar versiones menores (1.35 -> 1.36,
     no 1.35 -> 1.37)

COMO FUNCIONA EL UPGRADE INTERNAMENTE:
----------------------------------------

kubeadm upgrade apply hace en el control plane:
  1. Verifica compatibilidad de la nueva version
  2. Descarga las nuevas imagenes del control plane
  3. Actualiza los static pods (apiserver, etcd, etc)
     con las nuevas imagenes uno a uno
  4. Actualiza el kubeconfig y los certificados
  5. Actualiza los addons (kube-proxy, CoreDNS)

El upgrade de kubelet y kubectl es separado porque
no son gestionados por kubeadm, son paquetes apt.

TU ENTORNO DE PRACTICA:
  Version actual:  v1.35.5
  Version destino: v1.36.x (latest patch de 1.36)

PASO 1 - UPGRADE DEL CONTROL PLANE [CP]
-----------------------------------------

Actualizar el repositorio apt a la version 1.36:

  sudo sed -i \
    's|stable:/v1.35|stable:/v1.36|g' \
    /etc/apt/sources.list.d/kubernetes.list

  sudo apt-get update

Verificar versiones disponibles de kubeadm 1.36:

  apt-cache madison kubeadm | grep 1.36 | head -5

Quitar el hold de kubeadm para poder actualizarlo:

  sudo apt-mark unhold kubeadm

Instalar la nueva version de kubeadm:

  sudo apt-get install -y kubeadm=1.36.*

Volver a marcar kubeadm con hold:

  sudo apt-mark hold kubeadm

Verificar la nueva version de kubeadm:

  kubeadm version
  # esperado: v1.36.x

Verificar el plan de upgrade antes de ejecutarlo:

  sudo kubeadm upgrade plan

  La salida muestra:
  - Version actual del cluster
  - Version a la que se puede upgradear
  - Componentes que seran actualizados
  - Si hay algun problema que impide el upgrade

Ejecutar el upgrade del control plane:

  sudo kubeadm upgrade apply v1.36.0

  Este proceso tarda 3-5 minutos.
  Veras como cada componente se actualiza.
  Al final debe mostrar:
  "SUCCESS! Your cluster was successfully upgraded
   to v1.36.0"

Hacer drain del nodo control plane antes de
actualizar kubelet (evacua los pods del nodo):

  kubectl drain cka-cp1 \
    --ignore-daemonsets \
    --delete-emptydir-data

Quitar el hold de kubelet y kubectl:

  sudo apt-mark unhold kubelet kubectl

Actualizar kubelet y kubectl en el control plane:

  sudo apt-get install -y \
    kubelet=1.36.* \
    kubectl=1.36.*

Volver a marcar con hold:

  sudo apt-mark hold kubelet kubectl

Reiniciar kubelet:

  sudo systemctl daemon-reload
  sudo systemctl restart kubelet

Volver a poner el nodo en servicio:

  kubectl uncordon cka-cp1

Verificacion del control plane:

  kubectl get nodes
  # esperado:
  # cka-cp1       Ready   control-plane   v1.36.x
  # cka-worker1   Ready   <none>          v1.35.5
  # cka-worker2   Ready   <none>          v1.35.5

PASO 2 - UPGRADE DE WORKERS [W]
---------------------------------

Repetir para cada worker. Siempre de uno en uno.
Nunca upgradear dos workers simultaneamente en
produccion, el examen tampoco lo pide.

EJECUTAR EN cka-worker1:

Actualizar el repositorio apt a 1.36:

  sudo sed -i \
    's|stable:/v1.35|stable:/v1.36|g' \
    /etc/apt/sources.list.d/kubernetes.list

  sudo apt-get update

Quitar hold e instalar nueva version de kubeadm:

  sudo apt-mark unhold kubeadm
  sudo apt-get install -y kubeadm=1.36.*
  sudo apt-mark hold kubeadm

Actualizar la configuracion del nodo worker:

  sudo kubeadm upgrade node

  Este comando es diferente a kubeadm upgrade apply.
  En workers solo actualiza la configuracion local
  del kubelet, no los componentes del control plane.

Desde cka-cp1, hacer drain del worker:

  kubectl drain cka-worker1 \
    --ignore-daemonsets \
    --delete-emptydir-data

  Esto evacua todos los pods del worker antes
  de actualizarlo. Los pods se replanifican
  en otros nodos disponibles.

DE VUELTA EN cka-worker1:

Actualizar kubelet y kubectl:

  sudo apt-mark unhold kubelet kubectl
  sudo apt-get install -y kubelet=1.36.* kubectl=1.36.*
  sudo apt-mark hold kubelet kubectl

  sudo systemctl daemon-reload
  sudo systemctl restart kubelet

Desde cka-cp1, volver a poner el worker en servicio:

  kubectl uncordon cka-worker1

Verificacion tras upgradear worker1:

  kubectl get nodes
  # esperado:
  # cka-cp1       Ready   control-plane   v1.36.x
  # cka-worker1   Ready   <none>          v1.36.x
  # cka-worker2   Ready   <none>          v1.35.5

Repetir el mismo proceso para cka-worker2.

Verificacion final del cluster completo:

  kubectl get nodes
  # esperado:
  # cka-cp1       Ready   control-plane   v1.36.x
  # cka-worker1   Ready   <none>          v1.36.x
  # cka-worker2   Ready   <none>          v1.36.x

  kubectl get pods -A
  # todos los pods del sistema deben estar Running

TROUBLESHOOTING - CASOS REALES
--------------------------------------------------

CASO 1: kubeadm upgrade apply falla con error de etcd
------------------------------------------------------

Sintoma:
  error execution phase upgrade/etcd: couldn't
  upgrade control plane: timed out waiting for
  the condition

Causa mas comun:
  etcd no esta healthy antes del upgrade.

Diagnostico:

  sudo ETCDCTL_API=3 etcdctl endpoint health \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key

  crictl ps | grep etcd
  crictl logs <id-del-contenedor-etcd>

Solucion:
  Resolver el problema de etcd antes de reintentar
  el upgrade. Ver caso 3 de este modulo.

CASO 2: kubectl drain falla con pods que no se pueden evict
-----------------------------------------------------------

Sintoma:
  error: unable to drain node "cka-worker1",
  aborting command...
  There are pending nodes to be drained:
  cka-worker1
  error: cannot delete Pods with local storage

Causa:
  Hay pods con emptyDir volumes o pods sin
  un controller (pods standalone).

Solucion:
  Agregar los flags necesarios:

  kubectl drain cka-worker1 \
    --ignore-daemonsets \
    --delete-emptydir-data \
    --force

  --ignore-daemonsets: ignora pods de DaemonSets
                       que no pueden moverse
  --delete-emptydir-data: acepta perder datos
                          de volumes emptyDir
  --force: elimina pods sin controller

CASO 3: etcd no arranca tras restore
--------------------------------------

Sintoma:
  kubectl get nodes no responde.
  crictl ps | grep etcd no muestra etcd Running.

Diagnostico:

  crictl ps -a | grep etcd
  # ver el estado del contenedor aunque no este Running

  crictl logs <id-etcd>
  # buscar el error exacto

  sudo journalctl -u kubelet | tail -50
  # ver si kubelet reporta algun error al arrancar etcd

Causas comunes y soluciones:

  1. Permisos incorrectos en el directorio de datos:
     sudo chown -R root:root /var/lib/etcd
     sudo chmod 700 /var/lib/etcd

  2. El manifiesto de etcd apunta al directorio
     de datos anterior. Verificar:
     cat /etc/kubernetes/manifests/etcd.yaml \
       | grep data-dir
     # debe coincidir con donde restauraste los datos

  3. El snapshot esta corrupto:
     sudo ETCDCTL_API=3 etcdctl snapshot status \
       /opt/etcd-backup.db
     # si falla, el snapshot no es valido

CASO 4: nodo worker en estado NotReady tras upgrade
-----------------------------------------------------

Sintoma:
  kubectl get nodes muestra el worker NotReady
  despues de actualizar kubelet.

Diagnostico:

  kubectl describe node cka-worker1
  # buscar en la seccion Conditions el motivo

  En el worker:
  sudo systemctl status kubelet
  sudo journalctl -u kubelet | tail -30

Causas comunes:

  1. kubelet no se reinicio tras el upgrade:
     sudo systemctl daemon-reload
     sudo systemctl restart kubelet

  2. Version de kubelet no coincide con el cluster:
     kubelet --version
     # debe ser v1.36.x

  3. containerd no esta corriendo:
     sudo systemctl status containerd
     sudo systemctl restart containerd

CASO 5: static pod no aparece en kubectl get pods
--------------------------------------------------

Sintoma:
  Modificaste un manifiesto en
  /etc/kubernetes/manifests/ pero el pod no
  aparece o no se actualiza.

Diagnostico:

  ls -la /etc/kubernetes/manifests/
  # verificar que el fichero existe y tiene permisos

  sudo systemctl status kubelet
  # kubelet debe estar activo para leer los manifiestos

  crictl ps -a | grep <nombre-componente>
  # ver el estado real del contenedor

Solucion:
  kubelet detecta cambios en manifests automaticamente.
  Si no lo hace, forzar recarga:

  sudo systemctl restart kubelet

  Si el manifiesto tiene errores de sintaxis YAML,
  kubelet no podra crear el pod. Verificar:

  cat /etc/kubernetes/manifests/kube-apiserver.yaml \
    | python3 -c "import sys,yaml; yaml.safe_load(sys.stdin)"
  # sin output = YAML valido
  # con error = YAML invalido, revisar indentacion

--------------------------------------------------
CHECKPOINT - VERIFICACION DE COMPRENSION
--------------------------------------------------

Pregunta 1:
  Cual es el orden correcto para hacer un upgrade
  de v1.35 a v1.36 en un cluster con 1 CP y 2 workers?

Respuesta esperada:
  1. Upgrade kubeadm en el CP
  2. kubeadm upgrade apply v1.36.0 en el CP
  3. drain del CP
  4. Upgrade kubelet y kubectl en el CP
  5. uncordon del CP
  6. Para cada worker:
     upgrade kubeadm, kubeadm upgrade node,
     drain desde CP, upgrade kubelet,
     restart kubelet, uncordon desde CP

Pregunta 2:
  Por que no puedes hacer el restore de etcd
  mientras etcd esta corriendo?

Respuesta esperada:
  etcd bloquearia el directorio de datos y los
  nuevos datos restaurados entrarian en conflicto
  con el estado en memoria de etcd. El resultado
  seria corrupcion de datos o un restore que no
  tiene efecto. Debes detener etcd moviendo su
  manifiesto fuera de /etc/kubernetes/manifests/

Pregunta 3:
  Donde guarda kubeadm los certificados TLS
  del cluster y como verificas su expiracion?

Respuesta esperada:
  En /etc/kubernetes/pki/
  sudo kubeadm certs check-expiration

Pregunta 4:
  Que diferencia hay entre kubeadm upgrade apply
  y kubeadm upgrade node?

Respuesta esperada:
  kubeadm upgrade apply se ejecuta en el CP y
  actualiza todos los componentes del control plane
  (apiserver, etcd, scheduler, controller-manager)
  ademas de los addons (kube-proxy, CoreDNS).
  kubeadm upgrade node se ejecuta en los workers
  y solo actualiza la configuracion local de kubelet,
  no toca componentes del control plane.

Pregunta 5:
  Un pod standalone (sin Deployment ni ReplicaSet)
  esta corriendo en el nodo que quieres upgradear.
  kubectl drain falla. Como lo resuelves?

Respuesta esperada:
  kubectl drain <nodo> \
    --ignore-daemonsets \
    --delete-emptydir-data \
    --force
  El flag --force elimina pods sin controller.
  Hay que aceptar que ese pod no se replanificara
  automaticamente en otro nodo.
